# Prisma + NestJS + PostgreSQL — Agent Workflow

## The One Rule That Overrides Everything Else

**Never touch the `prisma/migrations/` directory.** Do not create folders inside it. Do not write `.sql` files inside it. Do not invent timestamps. The migration folder structure looks like `prisma/migrations/20240101120000_some_name/migration.sql` — but that timestamp is generated by Prisma CLI, not by you.

This is the most important constraint in this skill. Every other rule below flows from it.

---

## Why This Matters

When an AI agent hand-writes migration files, it:
- Invents a fake timestamp that may collide with or precede real migrations
- Bypasses Prisma's shadow database checks, which detect schema drift
- Corrupts the `_prisma_migrations` history table, making future `migrate deploy` fail
- Creates SQL that may not match what Prisma would actually generate (different column ordering, constraint names, index names, etc.)

The human developer controls migration creation. The agent controls only `schema.prisma`.

---

## The Four Hard Rules

### Rule 1 — Never hand-write migration files

❌ Do NOT create `prisma/migrations/<timestamp>_<name>/migration.sql`  
❌ Do NOT create any folder under `prisma/migrations/`  
❌ Do NOT suggest writing SQL migration content manually  

### Rule 2 — Never use `prisma db push` for normal schema changes

`prisma db push` is for throwaway prototyping only. It syncs schema without creating migration history. If you use it on a real project, you lose the ability to use `migrate deploy` in staging/production — forever, for that project.

❌ Do NOT suggest `bun x prisma db push` for anything other than an isolated sandbox experiment  
✅ The only schema-change command for this project is `bun x prisma migrate dev --name <descriptive_name>`

### Rule 3 — Keep ETL and backfill logic out of migration files

Migration files run automatically as part of deploy. Mixing data transformation logic into structural migrations causes:
- Non-idempotent deploys
- Unexpected data mutations on rollback/retry
- Impossible-to-review diffs

Data backfills and ETL scripts belong in separate, purpose-built scripts (e.g., `scripts/backfill-<name>.ts`) run manually or via a one-off job, never inside `migration.sql`.

### Rule 4 — Schema changes stop at `schema.prisma`

Your job ends at editing `schema.prisma`. After that, hand off to the human developer.

When a schema change is needed, you:
1. Edit `prisma/schema.prisma` with the desired model change
2. Stop
3. Print this exact block to the developer:

```
🛑 Schema updated. Run this command to create and apply the migration:

  bun x prisma migrate dev --name <descriptive_name>

Then run:

  bun x prisma generate

to regenerate the Prisma Client types.
```

Replace `<descriptive_name>` with a specific, snake_case name that describes the change (e.g., `add_refresh_token_to_user`, `make_email_unique`, `add_created_at_to_orders`).

## Breaking Change Warnings

Before making any of the following schema changes, warn the developer explicitly:

| Change | Why it's breaking | What to say |
|---|---|---|
| Renaming a column | Old column is dropped, data is lost | "⚠️ Renaming `oldName` → `newName` will DROP the old column. Prisma generates a DROP + ADD, not an ALTER. To preserve data, the migration SQL must be customized (use `--create-only` and manually write a `RENAME COLUMN` or copy data first)." |
| Making an optional field required (`?` removed) | Existing `NULL` rows will fail migration | "⚠️ Making `fieldName` required will fail if any existing rows have NULL values. Ensure a `@default(...)` is set, or provide a backfill before running the migration." |
| Changing a field's type | May require data casting | "⚠️ Changing the type of `fieldName` from `X` to `Y` may require a data cast in the migration SQL. Review the generated SQL before applying." |
| Dropping a model or field | Data permanently deleted | "⚠️ Dropping `ModelName.fieldName` will permanently delete that column's data. Confirm this is intentional." |
| Adding a unique constraint | Will fail if duplicate values exist | "⚠️ Adding `@unique` to `fieldName` will fail if there are duplicate values in the database. Check for and resolve duplicates first." |
| Adding a `@relation` with `onDelete: Cascade` | Can cascade-delete unexpected data | "⚠️ Cascade delete on this relation will remove related records automatically. Verify this is the intended behavior." |

For renaming specifically, the recommended safe approach is:
1. Add the new column (optional, with default)
2. Write a backfill script to copy data from old → new
3. Switch app code to use the new column
4. Drop the old column in a follow-up migration

## PrismaService Pattern (NestJS)

Use this pattern for the `PrismaService`. Do not deviate from it:

```typescript
// src/prisma.service.ts
import { Injectable, OnModuleInit } from '@nestjs/common';
import { PrismaClient } from '../generated/prisma/client.js';
import { PrismaPg } from '@prisma/adapter-pg';

@Injectable()
export class PrismaService extends PrismaClient implements OnModuleInit {
  constructor() {
    const adapter = new PrismaPg({
      connectionString: process.env.DATABASE_URL as string,
    });
    super({ adapter });
  }

  async onModuleInit() {
    await this.$connect();
  }
}
```

Register it in `AppModule` (or a shared `DatabaseModule`):

```typescript
@Module({
  providers: [PrismaService],
  exports: [PrismaService],
})
export class DatabaseModule {}
```

Inject it into feature services via the constructor — never instantiate `PrismaClient` directly in feature code.

## schema.prisma Configuration (PostgreSQL)

Always ensure this baseline config is present:

```prisma
generator client {
  provider = "prisma-client"
  output   = "../src/generated/prisma"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}
```

Key notes:
- `output` path keeps generated types inside `src/` where TypeScript resolution works cleanly
- Never hardcode `DATABASE_URL` — always use `env()`
- For connection pooling (PgBouncer), add `directUrl = env("DIRECT_URL")` alongside `url` and use `url` for app queries

## Query Patterns and Best Practices

**Use generated Prisma types for DTOs** — don't re-declare model shapes:

```typescript
// Good
async createUser(data: Prisma.UserCreateInput): Promise<User> {
  return this.prisma.user.create({ data });
}

// Avoid — redundant typing
async createUser(email: string, name: string): Promise<User> { ... }
```

**Transactions** — use `this.prisma.$transaction([...])` for multi-step operations that must be atomic:

```typescript
const [post, update] = await this.prisma.$transaction([
  this.prisma.post.create({ data: postData }),
  this.prisma.user.update({ where: { id }, data: { postCount: { increment: 1 } } }),
]);
```

**Soft deletes** — if the project uses soft deletes, add a `deletedAt DateTime?` field and filter with `where: { deletedAt: null }` in all queries. Do not use `delete()` on soft-delete models.

**Pagination** — use cursor-based pagination for large datasets, not `skip`/`take` offset pagination, which degrades at large offsets:

```typescript
// Cursor-based (preferred for large tables)
const posts = await this.prisma.post.findMany({
  take: 20,
  cursor: { id: lastId },
  skip: 1, // skip the cursor itself
  orderBy: { id: 'asc' },
});
```

**Select only what you need** — avoid returning full model objects when only a subset of fields is needed:

```typescript
const userEmails = await this.prisma.user.findMany({
  select: { id: true, email: true },
});
```

## Migration Deployment Workflow (Reference)

| Environment | Command | Notes |
|---|---|---|
| Development | `bun x prisma migrate dev --name <name>` | Creates + applies migration, regenerates client |
| CI/CD / Production | `bun x prisma migrate deploy` | Applies pending migrations only, non-interactive, safe |
| After code deploy | `bun x prisma generate` | Regenerate client if schema changed |
| Never in production | `prisma migrate dev` | Can prompt for DB reset |
| Never in production | `prisma db push` | No migration history |
| Never in production | `prisma migrate reset` | Drops and recreates the database |

`migrate deploy` uses PostgreSQL advisory locking — safe for concurrent deploys.

## Environment Variable Checklist

Make sure `.env` (or the deployment environment) includes:

```env
DATABASE_URL="postgresql://user:password@host:5432/dbname?schema=public"
# Optional: for PgBouncer / connection pooling setups
# DIRECT_URL="postgresql://user:password@host:5432/dbname?schema=public"
```

Never commit real credentials. Use `.env.example` with placeholder values.

---

## Common Mistakes to Avoid

- **Do not** run `bun x prisma generate` and skip `migrate dev` when you've changed `schema.prisma` — the client will be out of sync with the actual database schema
- **Do not** edit migration files that have already been applied — create a new migration instead
- **Do not** check in `.env` with real credentials — only `.env.example`
- **Do not** import from `@prisma/client` directly if the project uses a custom `output` path — use the project-specific import path (e.g., `../generated/prisma/client.js`)
- **Do not** call `this.prisma.$connect()` in every service — `OnModuleInit` in `PrismaService` handles it once at startup
